# Financial Admin Reports Proposal

## 1. Feature Overview
This proposal outlines the functionality for the **Financial Admin Reports** module within the Admin Dashboard. The goal is to provide actionable financial insights through sortable tables and exportable data, covering sales performance, order metrics, and item-level analysis.

## 2. Report Types

The system will support three primary report types, accessible via the "Reports" tab in the Admin Dashboard:

### A. Executive Summary
Provides a high-level aggregate view of financial performance for the selected period.
**Columns:**
*   **Orders:** Total number of valid orders (excluding Cancelled).
*   **Gross Sales:** Sum of item subtotals (pre-tax, pre-shipping).
*   **Discounts:** Total value of discounts applied.
*   **Shipping Costs:** Total shipping fees collected.
*   **Taxes (VAT):** Total VAT collected.
*   **Returns:** Total value of returned merchandise (Subtotal basis).
*   **Net Sales:** `Gross Sales` - `Discounts` - `Returns`.
*   **Total Collected:** Total transaction value (Gross + Shipping + Tax - Discounts).

### B. Sales Over Time (Daily)
Provides a daily breakdown of sales performance.
**Columns:**
*   **Date:** The date of the transaction (YYYY-MM-DD).
*   **Orders:** Number of orders placed on that date.
*   **Gross Sales:** Sum of item subtotals.
*   **Net Sales:** Gross Sales adjusted for discounts and returns.
*   **Shipping:** Shipping fees collected.
*   **Tax:** VAT collected.
*   **Total:** Total amount collected.

### C. Sales by Item
Analyzes performance at the product variant level.
**Columns:**
*   **Product SKU:** Unique identifier for the product variant.
*   **Product Name:** Name of the product (including variant details like Size/Color).
*   **Quantity Sold:** Total units sold.
*   **Revenue:** Gross revenue generated by this item (Unit Price * Quantity).

## 3. Functionality

### Date Filtering
*   **Start Date** and **End Date** pickers allow the admin to define the reporting period.
*   Defaults to the current month if not specified.

### Sortable Tables
*   All data tables generated in the frontend will be **sortable**.
*   Clicking a column header toggles between Ascending and Descending order.
*   Sorting logic supports:
    *   **Numeric values** (Currency, Counts).
    *   **Dates** (YYYY-MM-DD).
    *   **Text** (SKUs, Names).

### Export
*   A **"Export CSV"** button allows downloading the currently generated report as a `.csv` file.
*   The CSV export reflects the data filtered by the selected date range.

### Multi-Currency Handling
*   **Issue:** The current system allows different currencies per country, but aggregating them into a single "Total" is mathematically invalid without conversion.
*   **Proposed Solution:**
    *   Reports will group totals by **Currency** (e.g., separate rows for EUR, USD) OR convert all amounts to the Base Currency using a fixed exchange rate (if available).
    *   For the initial implementation, the report will display the **Currency Symbol** alongside the values, and distinct currencies will be flagged or grouped.

## 4. Technical Implementation Plan

The foundational infrastructure (Frontend UI and Backend API skeleton) is already present. The following actions are required to finalize the feature and ensure data accuracy.

### Backend Logic Refinement (`app/blueprints/api.py`)
The current implementation of `Net Sales` and `Returns` requires adjustment to ensure consistent accounting (Pre-tax vs Post-tax).

**Current Logic (Flawed):**
*   `Gross Sales` = Sum of `Subtotal` (Pre-tax).
*   `Returns` = Sum of `Total` (Post-tax/Shipping).
*   `Net Sales` = `Gross` - `Discounts` - `Returns` (Resulting in inconsistent subtraction).

**Proposed Logic (Corrected):**
*   **Returns:** Calculate as the sum of `Subtotal` (pre-tax) for orders with status `RETURNED`.
*   **Net Sales:** `Gross Sales` (Subtotal) - `Discounts` - `Returns` (Subtotal).
*   **Gross Sales:** Ensure this aggregates `subtotal_cents` for all non-cancelled orders.

### Frontend Verification
*   **Sorting:** Verify `app/static/js/admin_reports.js` correctly handles the sort order for all columns.
*   **UI:** Confirm `app/templates/admin.html` includes the necessary IDs (`report-type`, `report-start`, `report-end`, `btn-generate-report`) to bind with the JavaScript logic.

### Verification Strategy
1.  **Seed Data:** Create test orders with known values (e.g., 1 Normal, 1 Discounted, 1 Returned).
2.  **Generate Report:** Run the Executive Summary for the test period.
3.  **Validate Calculation:** Manually calculate the expected Net Sales and compare with the report output.
4.  **Test Export:** Download the CSV and verify the data matches the screen output.
